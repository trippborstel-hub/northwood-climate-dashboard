Reference File Library — the core IP

Six reference files that make the disclosure service repeatable. Together they encode the framework requirements, the voice, the interview protocol, and the quality gates. They are the assets that amortize across every engagement — the reason the service is fixed-price instead of hourly.

6 files · ~280 KB total · Generated iteratively, judged by output quality
Why reference files, not templates

Per Diego's Week 6 guidance: reference files balance over-specification (which stifles the AI's inference) and under-specification (which causes variability across runs). A template is too rigid — it produces identical boilerplate regardless of client. A loose prompt is too free — it produces a different disclosure every time.

These six files sit between those extremes. They name the structure, the datapoints, the voice, and the quality gates. They do not tell the AI what to say in any specific section. The content is inferred from the client's data; the reference files ensure it's inferred consistently and at publication quality.

The files are generated iteratively — draft, run against a test client, check output, refine. Their effectiveness is judged by the output, not by reading them. If the disclosures come out well, the files are working; if the output varies across runs, the files need tightening.

The Six Files
00Service Overview
Orientation · Every Step
The umbrella. Read first on every task. Orients the AI to what the service is, who the client is, the 7-step process, and what good vs. bad output looks like.
Key sections:
  • What we produce (TCFD + ESRS E1 as sister outputs from one process)
  • The client persona (mid-market PE, $1–5B AUM, ESG owned by COO/CFO)
  • The 7-step process in one place
  • What good output looks like: evidence-backed, honest about gaps, financially literate, concise, decision-useful
  • Scope boundaries — organizational boundary, reporting period, exclusions
Referenced by every subsequent reference file
01TCFD Pillar Template
Step 5 · Drafting
Structural guide for the four TCFD pillars at fund level. Caps length, names what each section must contain, provides "what to do if this pillar is thin" guidance for first-year disclosures.
What it specifies:
  • Document structure (About → Governance → Strategy → Risk Mgmt → Metrics & Targets → Forward Agenda → Appendices)
  • Target page counts per section (15–25 pages total)
  • Required content in each TCFD recommended disclosure
  • Scenario analysis expectations (SSP3-7.0 + SSP5-8.5 at 2030/2035)
  • First-year honesty guidance (governance gaps, no committed target, etc.)
Referenced in Step 5 — Draft Generation (TCFD)
02ESRS E1 Datapoint Map
Step 5 · Drafting
Datapoint-by-datapoint guide for the nine ESRS E1 disclosure requirements (E1-1 through E1-9). Unlike TCFD, ESRS is datapoint-driven — every DR has specific fields, and silence is non-compliance.
What it specifies:
  • What EFRAG asks for in each of E1-1 through E1-9
  • Required datapoints per DR (checkbox style)
  • PCAF methodology for Cat 15 financed emissions (standard for PE)
  • Double-materiality context requirements from ESRS 2
  • Phase-in reliefs available to voluntary preparers
  • Consistency requirements with the TCFD disclosure
Referenced in Step 5 — Draft Generation (ESRS E1)
03Interview & Questionnaire Script
Step 4 · Gap-Fill
The script for the AI-led 60-minute governance interview plus async portco questionnaires. Ten gap areas, each with specific questions, follow-ups, and an output template.
What it specifies:
  • Live vs. async protocol (governance topics live; portco data async)
  • Ten gap areas: board oversight, mgmt incentives, policy, targets, ICP, Scope 3, renewables, DD integration, LP cadence, resilience
  • Interview etiquette (one question at a time, paraphrase back, accept "no" answers)
  • Synthesis document template — sent to client for sign-off before drafting
  • Decisions-awaiting-client-sign-off section (no surprises in final review)
Referenced in Step 4 — Interviews & Questionnaires
04Human QA Checklist
Step 6 · Quality Gate
The checklist a senior human reviewer runs before any disclosure is marked final. Seven sections of hard pass/fail gates. This is where AI-generated novelty gets validated — zero AI participation in this step.
What it specifies:
  • A — Source Integrity: every number traces to source; emission factor vintages stated
  • B — Framework Compliance: all TCFD pillars / all 9 ESRS E1 DRs addressed
  • C — Honesty & Credibility: no unevidenced claims, no superlatives, gap-acknowledgment
  • D — Decision-Usefulness: could a new IC member get up to speed in 30 min?
  • E — LP Readability: can an LP DDQ coordinator answer from this?
  • F — Voice & Style: forbidden terms absent; concision
  • G — Adversarial Reading: greenwashing risk, forward-looking caveats
Referenced in Step 6 — Human QA (the "novelty validation" step)
05Voice and Style
Step 5 & 6 · Enforced
How a Kith disclosure reads. Inherits from the Kith master voice guide. Adds disclosure-specific conventions: when "commit" is allowed, how to talk about targets, how to frame scenarios, forbidden phrases.
Key rules:
  • "Commit" only for formally board-approved decisions with a date
  • "Science-aligned" not "SBTi-validated" unless actually validated
  • "Screening-level Scope 3" when partial, not "Scope 3 inventory"
  • Forbidden phrases list (no "we believe," "industry-leading," "going forward," etc.)
  • Sentence-level rules (active voice, delete throat-clearing, one idea per sentence)
  • Headings declarative not thematic ("Scope 2: 10,767 tCO2e" beats "Our Emissions Footprint")
Referenced in Step 5 (drafting) and Step 6 (QA Section F)

The Iteration Loop

Diego's Week 6 guidance applied

"I have never written a reference file manually. It's always a process of iteration. I judge the reference files by the output. It produces great output and keeps doing it — whatever's in there, it probably works very well."

These six files were generated iteratively with AI during the Week 6 build. Draft → run against Northwood's data → inspect output → identify where the output drifted from what a good disclosure looks like → refine the reference file → re-run. Not once in this process was a file hand-authored from a blank page.

When the output quality stabilizes across runs, the file is ready. When output varies meaningfully across runs on the same input, the file needs tightening. Effectiveness is measured in output quality, not file length or reader inspection.

Where the files live

Inside the working directory for this engagement, under northwood/disclosures/reference-files/. On production engagements they'd be copied from a master reference library, not recreated.

The production model: one master library in _reference/kith-disclosure-service/, copied into each engagement folder at kickoff. Improvements made during any engagement flow back to the master library after QA sign-off. This is the compounding-engagement model — every client makes the service better for the next client.
